## 出力要件 {#output-requirements}

- 完全なドキュメントを作成します。
- 次のガイドラインに従ってください。

## 声とトーン {#voice-and-tone}

- 機能自体を説明するのではなく、ユーザーが完了しようとしているタスクのために記述します。例えば、「APIキーを安全に保存するために変数を使用する」と記述し、「この機能はAPIキーを安全に保存できるように設計されています」とは記述しません。
- マーケティング用語を使用しないでください。例えば、「簡単に」、「強力に」、「単に」は使用しないでください。

## 記述スタイル {#writing-style}

- 現在形を使用します: 「システムは管理します」を使用し、「システムは管理するでしょう」は使用しません。
- 能動態を使用します: 「デベロッパーはコードを書きます」を使用し、「コードはデベロッパーによって書かれます」は使用しません。
- 米国式のスペルを使用します。
- 直接的に記述します: 「この機能を使用して...」を使用し、「これにより...が可能になります」または「これにより...が可能になります」は使用しません。
- 簡潔に記述します: 不要な単語を削除します。
- 行は100文字程度で分割します。リンクは分割しません。
- 可能な場合、文は20単語未満とします。複雑なアイデアは複数の文に分割します。
- 8年生の読み取りレベルを目指します。

## GitLab製品名 {#gitlab-product-names}

- 所有格は使用しません: 「GitLabのポリシー」ではなく「GitLab policies」
- 「GitLab Duo」を使用し、「Duo」は使用しません。
- 「GitLab Duo Agent Platform」を使用し、「DAP」または「Duo Agent Platform」は使用しません。
- 製品:
  - 「GitLab.com」を使用し、「GitLab SaaS」は使用しません。
  - 「GitLab Self-Managed」を使用し、「Self-managed」は使用しません。
  - 「GitLab Dedicated」を使用し、「Dedicated」は使用しません。
  - 「政府機関向けGitLab Dedicated」を使用し、「Dedicated for Government」は使用しません。

## 大文字と小文字 {#capitalization}

- トピックのタイトル: 文頭大文字を使用します。
- UIテキスト: インターフェースの正確な大文字と小文字に一致させます。
- 機能名: 小文字を使用します。

## テキストの書式設定 {#text-formatting}

- UI要素（ボタン、メニュー、ページ、設定）のみを太字（**text**）にします。
- コマンドラインインターフェース、ファイル名、変数、キーワードにはインラインコード（`text`）を使用します。
- キーボード入力は次の形式を使用します: <kbd>Control</kbd>+<kbd>C</kbd>。
- コマンドラインインターフェースコマンドおよび複数行コードにはコードブロックを使用します。適切な言語識別子を使用します。
  ```shell
  git commit -m "message"
  ```

## リスト {#lists}

- 順序なしリストには「-」、順序付きリストのすべての項目には「1.」を使用します。
- 1つのステップのみを持つタスクには、順序なしリスト項目を使用します。
- 項目を並列にします。
- 各行を大文字で始め、ピリオドで終わらせます。
- 各リストの前後に空行を使用します。

## リンク {#links}

- 同じリポジトリ内のリンクには相対パスを使用します。例: `[text](path/to/file.md)`。
- 「ここ」ではなく、説明的なリンクテキストを使用します。
- イシューリンクに番号を含めます。例えば、「詳細については、イシュー12345を参照してください。」
- 「詳細については、`[link text](<link>)`を参照してください。」のスタイルに従ってください。
- 同じ段落内に複数のリンクを含めないでください。

## 見出し {#headings}

- \## から #### (H2-H4) を使用します。レベルを飛ばさないでください。
- H1は使用しないでください。フロントマターのタイトルがH1です。
- 文頭大文字を使用します。太字は使用しないでください。
- 一般的なタイトルは避けてください。代わりに、次を行えます。
  - 概念および参照セクションでは、概念または参照が何であるかを説明する記述的な名詞を使用します: 「アクセス制御」、「グループ階層」、「保護レベル」、「保護レベルを理解する」
  - 手順セクションでは、ユーザーが何を達成するかを説明する動詞を使用します: 「グループを作成する」、「グループからメンバーを削除する」、「プロジェクトでフローを表示する」
  - 「概要」、「はじめに」、「設定」、「構成」、「使用方法」などの曖昧なタイトルは使用しないでください。

## 句読点 {#punctuation}

- オックスフォードカンマ（すべてのリスト項目の間にあるカンマ）を使用します。
- セミコロン、エムダッシュ、カーリークオートは避けてください。
- プレースホルダーテキストまたは変数には、<your_value>を使用します。

## ナビゲーションステップ {#navigation-steps}

- 場所を最初に、次にアクション: 「左サイドバーで、**設定**を選択します。」
- 簡潔に記述します: 「**保存**」を使用し、「変更を有効にするには**保存**を選択します」は使用しません。
- オプションのステップは「オプション。」で始めます。
- UIには、トップバー、左サイドバー、右サイドバー、および詳細パネルを使用します。

## テーブル {#tables}

- Description列は右側に配置します。
- ヘッダーには文頭大文字を使用します。
- 機能テーブルには、{{< yes >}}および{{< no >}}のショートコードを使用します。

## アラート {#alerts}

- アラート（ノートや警告など）は控えめに使用します。
- 次の構文を使用します:
  ```markdown
  > [!note]
  > This is something to note.
  ```
- アラートは次の目的で使用します:
  - **Warnings**: 重大な結果を伴う破壊的なアクション、セキュリティ上の影響、または失敗を引き起こす可能性のある重要な制約。
  - **ノート**: コンテキスト、例外、または特殊なケースを明確にする補足情報。

## 避けるべき一般的な間違い {#common-mistakes-to-avoid}

- 「There is」または「There are」は避けてください。「パイプラインにエラーがあります」を使用し、「There are errors in theパイプライン」は使用しません。
- 「it」ではなく、特定の名詞を使用します。
- 「-ing」の単語ではなく、能動態の動詞を使用します。
- ラテン語の略語は避けてください。「e.g.」の代わりに「for example」を使用します。「via」の代わりに「through」または「by using」を使用します。
- 「this powerful feature」、「easily」、「simply」などの余分なフレーズは避けてください。何がどのように機能するかを具体的に記述します。

### 繰り返し {#repetition}

- 同じページ内またはリンクされているトピックで以前に説明された情報を繰り返さないでください。
- 各セクションは新しい情報を追加する必要があります。説明したばかりの内容を要約しないでください。
- 最初の段落でタイトルや紹介文を繰り返さないでください。

### スコープ {#scope}

- 単一の概念、用語、または手順ステップのために新しいページを作成しないでください。

### 正確性 {#accuracy}

- 既存のコードベース、リンクされたドキュメント、またはすでにページにある内容に基づいた情報のみを含めます。
- 機能がどのように機能するかを推測したり、推定したりしないでください。
- コマンドラインインターフェース構文、API変数、またはUI要素名を勝手に作成しないでください。
